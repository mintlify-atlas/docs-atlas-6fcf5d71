A model in LoopBack 4 describes the shape of your domain data—its properties, types, and constraints. Models are plain TypeScript classes decorated with `@model` and `@property` to attach metadata used for validation, OpenAPI spec generation, and database mapping. Behavior like CRUD operations lives in [repositories](/data-access/repositories), keeping models focused on structure.

<Note>
A single model can be used with multiple different repositories, and models can be exchanged over REST, stored in databases, or mapped to OpenAPI schemas—all from the same class definition.
</Note>

## Model types

LoopBack 4 provides three base classes for domain objects:

<AccordionGroup>
  <Accordion title="Entity">
    An `Entity` has an identity property (`id`) that uniquely identifies each instance. Two `Entity` instances with the same `id` represent the same record. Use `Entity` for anything you persist in a database.

    ```typescript
    import {Entity, model, property} from '@loopback/repository';

    @model()
    export class Customer extends Entity {
      @property({id: true})
      id: string;

      @property()
      name: string;

      constructor(data?: Partial<Customer>) {
        super(data);
      }
    }
    ```
  </Accordion>
  <Accordion title="Model (value object)">
    A `Model` has no identity—equality is based on the values of its properties. Use `Model` for embedded or composite value objects like an address or a coordinate pair.

    ```typescript
    import {Model, model, property} from '@loopback/repository';

    @model()
    export class Address extends Model {
      @property() streetNum: number;
      @property() streetName: string;
      @property() city: string;
      @property() zipCode: string;

      constructor(data?: Partial<Address>) {
        super(data);
      }
    }
    ```
  </Accordion>
</AccordionGroup>

## The @model decorator

Apply `@model()` to a class to register it with the LoopBack metadata system. Without this decorator, `@property` metadata is ignored and no JSON schema is generated.

You can pass a `ModelDefinitionSyntax` object to configure model-level settings:

```typescript
@model({
  settings: {
    hiddenProperties: ['password'],
    strict: true,
  },
})
export class User extends Entity {
  // ...
}
```

<AccordionGroup>
  <Accordion title="Model settings reference">
    | Setting | Type | Default | Description |
    |---|---|---|---|
    | `settings.description` | string | — | Human-readable description of the model |
    | `settings.forceId` | boolean | `true` | Prevents clients from setting auto-generated IDs manually |
    | `settings.hiddenProperties` | string[] | — | Properties omitted from `.toJSON()` output |
    | `settings.scope` | object | — | Default filter applied to every repository query |
    | `settings.strict` | boolean \| `"filter"` | `true` | Whether to reject properties not defined on the model |
  </Accordion>
</AccordionGroup>

## The @property decorator

Use `@property()` on each class member to define its type and constraints. LoopBack infers the TypeScript type automatically, but you can supply explicit options:

```typescript
@model()
export class Product extends Entity {
  @property({
    type: 'number',
    id: true,
    generated: true,
  })
  id: number;

  @property({
    type: 'string',
    required: true,
    jsonSchema: {
      minLength: 2,
      maxLength: 100,
    },
  })
  name: string;

  @property({
    type: 'number',
    default: 0,
  })
  stock?: number;
}
```

<AccordionGroup>
  <Accordion title="Property options reference">
    | Key | Required | Type | Description |
    |---|---|---|---|
    | `type` | Yes | string | Property type: `'string'`, `'number'`, `'boolean'`, `'date'`, `'object'`, `'array'`, `'any'` |
    | `id` | No | boolean | Marks the property as the primary key |
    | `generated` | No | boolean | ID value is generated by the database |
    | `required` | No | boolean | Field must be present in POST/PUT request bodies |
    | `default` | No | any | Default value when none is supplied |
    | `defaultFn` | No | string | Name of a built-in function to generate defaults: `"guid"`, `"uuid"`, `"uuidv4"`, `"now"` |
    | `hidden` | No | boolean | Excludes the property from `.toJSON()` response output |
    | `index` | No | boolean \| object | Creates a database index; use `{unique: true}` for unique constraints |
    | `description` | No | string | Documentation string for the property |
    | `jsonSchema` | No | object | Additional JSON Schema keywords for validation |
  </Accordion>
</AccordionGroup>

## Real example: the Todo model

The following model is taken directly from `examples/todo/src/models/todo.model.ts`:

```typescript
import {Entity, model, property} from '@loopback/repository';

@model()
export class Todo extends Entity {
  @property({
    type: 'number',
    id: true,
    generated: false,
  })
  id?: number;

  @property({
    type: 'string',
    required: true,
  })
  title: string;

  @property({
    type: 'string',
  })
  desc?: string;

  @property({
    type: 'boolean',
  })
  isComplete?: boolean;

  @property({
    type: 'string',
  })
  remindAtAddress?: string;

  @property({
    type: 'string',
  })
  remindAtGeo?: string;

  constructor(data?: Partial<Todo>) {
    super(data);
  }
}

export interface TodoRelations {
  // describe navigational properties here
}

export type TodoWithRelations = Todo & TodoRelations;
```

## Array properties

JavaScript arrays carry no type information, so LoopBack requires the `@property.array()` decorator for array properties:

```typescript
@model()
class Order extends Entity {
  @property.array(String)
  tags: string[];

  @property.array(Product)
  items: Product[];
}
```

You can pass a second argument to `@property.array()` for additional options, including `jsonSchema` constraints on individual items:

```typescript
@model()
class Customer extends Entity {
  @property.array(String, {
    jsonSchema: {
      format: 'email',
      minLength: 5,
      maxLength: 100,
    },
  })
  emails?: string[];
}
```

## Model inheritance

Extend an existing model to add or override properties:

```typescript
@model()
export class TimestampedEntity extends Entity {
  @property({type: 'date', defaultFn: 'now'})
  createdAt: Date;

  @property({type: 'date', defaultFn: 'now'})
  updatedAt: Date;
}

@model()
export class Product extends TimestampedEntity {
  @property({type: 'number', id: true, generated: true})
  id: number;

  @property({type: 'string', required: true})
  name: string;
}
```

<Note>
In LoopBack 4, model inheritance uses standard TypeScript `extends`. There is no `base` setting as in LoopBack 3.
</Note>

## Hidden properties

Mark sensitive properties so they are stored in the database but stripped from API responses:

```typescript
@model({
  settings: {
    hiddenProperties: ['password'],
  },
})
export class User extends Entity {
  @property({id: true}) id: number;
  @property({type: 'string'}) email: string;
  @property({type: 'string'}) password: string;
}
```

Alternatively, use the `hidden` option directly on the property:

```typescript
@property({type: 'string', hidden: true})
password: string;
```

## Generating models with the CLI

<Steps>
  <Step title="Run the model generator">
    ```bash
    lb4 model
    ```
  </Step>
  <Step title="Choose model type">
    Select **Entity** if the model will be persisted in a database, or **Model** for a value object.
  </Step>
  <Step title="Add properties">
    The CLI prompts you for each property's name, type, whether it's required, and any default value.
  </Step>
  <Step title="Review the output">
    The generated class is placed in `src/models/` and exported from `src/models/index.ts`.
  </Step>
</Steps>

## JSON schema generation

LoopBack generates JSON schemas automatically from model metadata. Use `getJsonSchema` from `@loopback/repository-json-schema` to access the schema programmatically—useful for validation or documentation:

```typescript
import {model, property} from '@loopback/repository';
import {getJsonSchema} from '@loopback/repository-json-schema';

@model()
class Product {
  @property({required: true})
  name: string;

  @property()
  price: number;
}

const jsonSchema = getJsonSchema(Product);
```

The call above returns:

```json
{
  "title": "Product",
  "properties": {
    "name": {"type": "string"},
    "price": {"type": "number"}
  },
  "required": ["name"]
}
```

LoopBack also uses these schemas automatically to validate incoming request bodies in your REST controllers.

## Custom table and column names

If your database table or column names differ from your model class names, configure them through connector-specific settings:

```typescript
@model({
  settings: {
    postgresql: {schema: 'public', table: 'products'},
  },
})
export class Product extends Entity {
  @property({
    type: 'number',
    id: true,
    postgresql: {
      columnName: 'product_id',
      dataType: 'integer',
    },
  })
  id: number;

  @property({
    type: 'string',
    postgresql: {columnName: 'product_name'},
  })
  name: string;
}
```

<Tip>
Check your connector's documentation for the full list of supported `dataType` values and connector-specific mapping options.
</Tip>
